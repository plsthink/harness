GOAL: Self-improve the harness unattended over a long gnhf session by working four goal-phases in a fixed order, advancing to the next phase only when the current one runs dry. Splitting by forcing function — rather than mixing all concerns in one open-ended objective — is what keeps the run producing real work instead of marginal churn.

PHASES (fixed order; each phase's full GOAL/PROCEDURE/FORCING lives in its own objective file — read that file for the active phase and follow its GOAL/PROCEDURE/FORCING. The ordering rationale below is umbrella-only):
1. DOGFOOD — docs/gnhf/dogfood-friction.md. First because it is the cheapest, highest-signal forcing function, the fixtures are ready, and it surfaces the defects and capability gaps that seed phases 2-3. Phase 1 fires only on an actual skill/product stumble; a skill that runs clean but whose lens merely reveals pre-existing drift in a fixture's own docs is a Phase-4 coherence finding, not Phase-1 friction — route it there.
2. UPSTREAM — docs/gnhf/upstream-integration.md. Before capability so you integrate external material before building internal and do not reinvent what upstream already offers.
3. CAPABILITY — docs/gnhf/capability-expansion.md. Builds the gaps phases 1-2 revealed.
4. COHERENCE — docs/gnhf/coherence-audit.md. Last because it operates on the churn the prior phases produced and is self-limiting — the natural terminator. Do NOT homogenize a member whose difference from its siblings is intentional-by-design.

LOOP MECHANICS: Read this run's notes.md to find the current phase (the phase named in the most recent iteration; default to phase 1, dogfood, on the first iteration). Do the current phase's next unit of forcing-function work by reading that phase's objective file (linked above) and following ITS GOAL/PROCEDURE/FORCING — but IGNORE that file's own HALT and RULES sections: they govern only a standalone single-goal run. Under the umbrella, a phase that finds no defensible forcing function does not stop the run; it means ADVANCE THE POINTER, and the run halts only per the HALT condition below (a full four-phase pass with zero change). Follow the shared rules in docs/gnhf/_rules.md. Name the concrete trigger and the current phase in your iteration summary. If the current phase has no defensible forcing function this iteration, advance the pointer to the next phase (wrapping coherence -> dogfood), note the advance in your summary, and that advance is the iteration. You normally advance one phase per iteration. Exception: if you can PROVE the remaining phases this cycle are all dry, you may advance through them in this one iteration and report HALT — proof means commit history shows no change since the last verified-dry sweep AND each remaining phase's own mechanical check confirms dry (upstream: git ls-remote resolves every pinned source to its recorded pin; capability: pipeline.md still spans the lifecycle; coherence: check-refs passes and no new churn exists). Proof is mandatory per phase; never fast-forward on assumption.

HALT: when you complete a full pass through all four phases without making a single change — every phase reporting no defensible forcing function in turn — report that the run's configured stop condition is met. A dry full cycle means the repo is improved out; halting beats manufacturing marginal churn.

RULES — full text and rationale in docs/gnhf/_rules.md (read and follow it). The two phase-specific deltas the umbrella adds:
1. ONE FORCING FUNCTION PER ITERATION — name the trigger AND the current phase; a phase with no defensible trigger advances the pointer instead of forcing a change.
2. HALT HONESTLY — for the umbrella the stop condition is a full four-phase cycle with zero change (see HALT/LOOP MECHANICS), not a single phase running dry.
