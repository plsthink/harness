GOAL: Improve the harness by reconciling one concrete internal inconsistency, trimming one concrete redundancy, or correcting one concrete leanness (authoring-standard) violation — smaller, more precise context performs better. This is the self-limiting goal — its forcing function runs out, and when it does the correct move is to halt, not to manufacture marginal churn. The halt discipline here is the strictest of all the prompts.

PROCEDURE: Run docs/scripts/check-refs.sh and the dev-script self-tests. For the domain-doc and conventions surface, find the defect by following the docs-review skill's lens against the harness's own docs — its structured sweep catches drift that ad-hoc manual reading misses (rule 6) — and check cites/leanness directly. Read the product surface for ONE real defect: a dangling or malformed cite, a term used two ways versus the docs/CONTEXT.md glossary, a duplicated spec that violates cite-don't-restate, a stale stance that contradicts current behavior or another recorded stance, or a unit that violates a NAMED ${CLAUDE_PLUGIN_ROOT}/shared/authoring-standard.md rule (a SKILL.md well over the ~100-line soft target, a description over its 1024-char budget, detail nested more than one level deep, shared content copied rather than cited by path, a fact restated rather than cited). The product surface includes the mocks/ fixture docs: a stale fixture glossary term, or a fixture entrypoint that omits an on-disk artifact, is a coherence defect like any in the harness's own docs. It ALSO includes the skill/agent INSTRUCTION layer that docs-review's lens does not reach (docs-review sweeps domain docs + conventions, never skills/ or agents/): trace one cross-skill/agent execution path (e.g. execute-issue -> builder -> tdd), or read one SKILL.md/agent clause against the reference or sibling contract it depends on, and reconcile a contract mismatch there — a clause that compresses its cited rule and drops a load-bearing distinction (branch vs checkout, "each" vs "each mutating"), two units stating the same contract differently (a tdd author gate the reviewer gate does not mirror), or a spec attributing behavior to a unit whose own procedure does not perform it. check-refs cannot catch these — the cite resolves — so they need reading the clause against its source, the highest-signal defect class this run surfaces. Fix the root cause. Re-run the checks.

FORCING FUNCTION: the specific defect, e.g. "skill X cites shared/Y.md with a bare backtick, violating authoring-standard's plugin-root-cite rule", "diagnose's Reads footer names a glossary path that no longer exists", or "skill X's SKILL.md inlines a 60-line decision table that belongs in references/, well past the ~100-line soft target". Name the file and the defect in your iteration summary.

HALT — the expected, correct outcome once the surface is clean: check-refs passes, the self-tests pass, and you find no genuine contradiction. Report that the run's configured stop condition is met. EXPLICITLY DO NOT homogenize a member that differs from its siblings when that difference is intentional-by-design: a divergent member is a defect only if it contradicts a recorded decision (a stance) or a stated single-source. Documented layering, justified deviations, and supersets are not defects. Leanness is bound to the standard the same way: a unit that breaks a NAMED authoring-standard rule is a defect, but prose that is merely longer than you would prefer — breaking no named rule — is NOT, so do not churn it smaller. Halting beats churning.

RULES — full text and rationale in docs/gnhf/_rules.md (read and follow it):
1. ONE FORCING FUNCTION PER ITERATION — name the concrete trigger in your summary.
2. HALT HONESTLY — never invent a trigger; the honest stop is the correct outcome.
3. MEASURE FRESH — product files aren't live until `claude plugin update harness@harness`.
4. Honor conventions/templates/authoring-standard; one coherent, committed change per iteration.
5. START BY READING docs/AGENTS.md and this run's notes.md before acting.
6. DOGFOOD VIA THE FITTING SKILL — follow a fitting skill's procedure instead of hand-rolling; skip only when none fits.
